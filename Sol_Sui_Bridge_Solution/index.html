Error Handling: rollback_epoch

When cross-chain bridging fails, Sui Vault encounters an error, or other unrecoverable issues occur, the Operator must roll back the Epoch and return funds to users.

Trigger Conditions
1. CCTP bridge failed — USDC remains in Solana usdc_vault (burn did not succeed)
2. CCTP bridge succeeded but Sui operation failed — USDC reached Sui pool address but failed to deposit into Vault
3. Sui Vault settle error — settle timeout or rejection
4. Other situations where Operator determines rollback is necessary
rollback_epoch Instruction Design
// Operator multisig (recommended 3/3 full sign) pub fn rollback_epoch( epoch_id: u64, reason: String, // Rollback reason (recorded on-chain for audit) refund_usdc: u64, // Actual USDC amount available for refund ) { // Precondition: Epoch status ∈ {Frozen, Bridging, Settling} // Cannot roll back Open or already Settled Epochs // Deposit scenario: Return user's pending_deposit USDC for user in epoch.deposit_users { user.claimable_usdc += user.pending_deposit; // Move to claimable user.pending_deposit = 0; } // Redeem scenario: Return user's pending_redeem_shares for user in epoch.redeem_users { user.confirmed_shares += user.pending_redeem_shares; // Shares restored user.pending_redeem_shares = 0; } epoch.status = EpochStatus::Failed; }

Share Pricing & NAV Audit Mechanism

The total_shares_received passed by the Operator during confirm_deposit_settle determines how many shares users receive. This value depends on the Sui Vault's NAV at the time. To prevent manipulation, a verification mechanism is needed.

Approach: Off-chain Audit + On-chain Constraints
1. On-chain — share_price sanity check (VaultMirror-A)
VaultMirror-A maintains a last_known_share_price field
At confirm_settle, calculates implied share_price = total_deposit_usdc / total_shares_received
If deviation from last_known_share_price > max_price_deviation (e.g., 5%), transaction reverts
On success, updates last_known_share_price

2. Off-chain — Operator provides proof materials
Operator must record before confirm: Sui Vault settle tx_hash, on-chain NAV snapshot at settle time
This data is stored in off-chain audit logs (or IPFS) for post-hoc verification

3. Time window — Prevent stale data
confirm_settle must complete within max_settle_duration (e.g., 72h) after freeze_epoch
After timeout, Epoch automatically enters rollback-eligible state (Operator must manually rollback or confirm)
VaultMirror-A Extended Fields
pub struct VaultMirrorA { // --- Base fields (original design) --- pub authority: Pubkey, // multisig authority pub usdc_vault: Pubkey, // USDC custody PDA pub current_epoch_id: u64, pub total_shares: u64, // Sum of all user shares pub paused: bool, // --- v5 new: NAV protection --- pub last_known_share_price: u64, // Last settle share_price (6 decimals) pub max_price_deviation_bps: u16, // Max allowed deviation (basis points, e.g., 500 = 5%) pub max_settle_duration: i64, // Max time from freeze to confirm (seconds) }

usdc_vault Fund Flow & Multi-Epoch Coexistence

usdc_vault is a unified USDC custody account that may simultaneously hold funds from multiple Epochs. Here is the fund isolation approach.

Core Principle: Logical Isolation, Physical Sharing
usdc_vault is a single PDA Token Account with one physical balance.
Each Epoch tracks its own USDC inflow/outflow via its SettleEpoch account, achieving logical isolation.

usdc_vault balance composition:
balance = Σ(unbridged Epoch deposit USDC) + Σ(bridged-back redeem USDC pending claim) + Σ(rollback refunds)

Invariant check:
After each Operator operation, the contract should verify:
usdc_vault.balance ≥ Σ(all users' claimable_usdc) + Σ(Open Epoch pending_deposit)
If not satisfied → transaction reverts

Same Epoch with Both Deposits and Redeems

A single Epoch may contain both user deposits and other users' redemptions. The strategy for mixed handling needs consideration.

StrategyDescriptionEvaluation
Option 1: Netting
Recommended		 Net deposits against redeems within an Epoch. E.g., deposit 3000 USDC + redeem 1000 shares can partially offset → only bridge the net amount cross-chain. Reduces cross-chain transfers and fees, but Ledger Program logic is more complex; requires precise calculation of net_deposit_usdc and net_redeem_shares.
Option 2: Separate Processing Deposits and redeems follow independent confirm flows. Each Epoch's deposits and redeems are settled separately. Simpler logic, but more cross-chain transfers. Suitable for initial version, can upgrade to Netting later.
Recommendation
Initial (v1): Separate Processing — confirm_deposit_settle and confirm_redeem_settle called independently, reducing development complexity.
Later (v2): Introduce Netting — When transaction volume grows, Netting significantly reduces cross-chain costs (~$1-5 gas + wait time per transfer).

UserAccount Structure (Complete)

// PDA seeds: [b"user", vault_mirror.key, wallet.key] pub struct UserAccount { pub wallet: Pubkey, // User's Solana wallet (Crossmint) pub vault_mirror: Pubkey, // Associated VaultMirror-A // --- Fund state --- pub pending_deposit: u64, // USDC deposited into usdc_vault but not yet settled pub pending_deposit_epoch: u64, // Epoch ID that pending_deposit belongs to pub confirmed_shares: u64, // Confirmed Sui Vault shares pub pending_redeem_shares: u64, // Shares submitted for redeem but not yet settled pub pending_redeem_epoch: u64, // Epoch ID that pending_redeem belongs to pub claimable_usdc: u64, // USDC available for withdrawal (after settle or rollback) // --- Historical stats --- pub total_deposited: u64, // Cumulative USDC deposited pub total_redeemed: u64, // Cumulative USDC redeemed pub total_shares_ever: u64, // Cumulative shares received pub created_at: i64, pub last_action_at: i64, }
Constraint: One Unsettled Epoch at a Time
A user's pending_deposit and pending_redeem_shares can each only belong to one Epoch.
If a user has pending_deposit in Epoch N, they cannot deposit again in Epoch N+1 (must wait for Epoch N to settle or rollback).

Why?
· Simplifies share allocation logic: no need to track user pending state across multiple Epochs
· Reduces contract complexity: UserAccount only needs to record one pending_deposit_epoch

UX impact:
· If Epoch cycle is short (e.g., daily freeze), the restriction is acceptable
· If Epoch cycle is long, may limit user flexibility → can consider supporting multi-Epoch pending later
